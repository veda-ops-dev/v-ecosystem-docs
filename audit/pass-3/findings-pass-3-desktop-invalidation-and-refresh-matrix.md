Checklist results
QM-1: PARTIAL — The doc defines trigger events rather than a query/response call chain. For this doc's type — a runtime event-to-consequence matrix — the event trigger model serves as the entry point specification: every event entry defines the condition that fires it. The doc states it is the "canonical reference for every event that triggers context invalidation or refresh." However, the mechanism by which the desktop application detects events — polling vs. subscription, push vs. pull, the refreshCoordinator role — is delegated: "when implementing refreshCoordinator event handling." The call chain from event source to refreshCoordinator to surface propagation is not defined in this doc.
QM-2: FAIL — No input parameters are defined. The matrix defines trigger conditions and consequences but specifies no input fields, data types, or valid values for any event trigger. Trigger descriptions are behavioral ("A new governing decision is written to Project V records") not parametric. No event payload schema is defined.
QM-3: FAIL — No response structure is defined. Each matrix entry specifies UI and LLM consequences in prose, not as a typed response record. No fields, types, or structured payload definitions exist for any event's consequence output.
QM-4: FAIL — No result limits or pagination are defined. The Sequencing Rules address event ordering priority but define no volume constraints for event queuing, batch processing, or concurrent handling limits.
QM-5: FAIL — No filter options are documented. The matrix is exhaustive by design. No mechanism for subscribing to or filtering a subset of events is defined.
FR-1: PARTIAL — A freshness model exists for evidence events. EVENT-08 fires when evidence "has exceeded the typical validity window for its evidence type." EVENT-09 defines when the staleness marker is removed. Multiple events specify that context must be reloaded before Class B/C gates. However, "the typical validity window for its evidence type" is not defined in this doc — it is deferred to VEDA-side docs — leaving the triggering threshold for EVENT-08 incomplete here.
FR-2: FAIL — No staleness threshold is defined. EVENT-08 fires when evidence has "exceeded the typical validity window" but no numeric or temporal value is stated. Decision context staleness definitions are in desktop-governance-and-gating-model.md, not here. The matrix defines what happens after staleness is detected, not when staleness begins.
FR-3: FAIL — No last_updated or equivalent timestamp field is defined on any event record or surface update payload. The Cross-Event Rule "Dismiss and Acknowledge Logging" requires timestamps on dismissal events but specifies no structured timestamp field on event records. EVENT-07 requires the context strip to show a "capture timestamp" for loaded evidence records, but that is a display requirement for evidence records, not a timestamp field on the event itself.
AL-1: FAIL — Activity trail logging is not specified for any event in the matrix. The Cross-Event Rule "Dismiss and Acknowledge Logging" states "every operator action that dismisses or acknowledges an invalidation or integrity item must be logged as a governance-relevant fact with timestamp and operator identity" but specifies this goes into "the session audit record" — not the ecosystem activity trail. EVENT-05 (Approval Written — Class B or C action completed) corresponds to approval.decide in the activity-trail-model.md vocabulary but makes no reference to that requirement. These are treated as different logging systems without any connection defined.
AL-2: FAIL — No dot-namespaced action field value is specified for any event. The activity-trail-model.md defines approval.decide, approval.request, approval.escalate as applicable action types. EVENT-05 describes a completed approval event but specifies no action vocabulary from the ecosystem activity trail model. No activity trail action vocabulary appears anywhere in the matrix.
AL-3: FAIL — No activity record fields are mentioned in the matrix. The Cross-Event Rule "Dismiss and Acknowledge Logging" mentions timestamp and operator identity for the session audit record, but does not reference the canonical required fields from the activity-trail-model.md (activity_id, actor_type, actor_system, entity_type, entity_id, target_system, action_class, cost fields, etc.).
DG-1: PARTIAL — EVENT-10 (Session Token Expiry or Runtime Scope Breakage) defines the degraded mode for the most severe failure case: all nine context categories become invalid, all surfaces enter blocked posture, Level 1 sequencing priority supersedes all other in-flight events, and a modal blocks further work. However, backend service unavailability while the session token remains valid — for example, Project V records temporarily unreachable during a context reload triggered by EVENT-01 — is not addressed as a degraded event. Partial failure scenarios have no defined degraded mode.
DG-2: PARTIAL — For EVENT-10, caller behavior during the degraded mode is fully specified: all Class B and C actions blocked, no governance-sensitive LLM outputs, re-initialization required before continuation. For backend partial unavailability and individual context category reload failure, caller behavior is not defined. Several events (EVENT-17, EVENT-05) require context reloads before gates are presented but do not define behavior if those reloads fail.
DG-3: PARTIAL — EVENT-10 specifies a recovery path: [Re-initialize Session] control, full reload of all context categories from governed records on re-init. No recovery path is defined for partial degradation (individual backend unavailable, individual category failing to reload). EVENT-09 defines an evidence refresh path but not a recovery path for an evidence refresh failure.
CA-1: FAIL — The doc does not state whether any event or event-handling operation incurs token cost or API cost. Events involving evidence API calls (EVENT-07, EVENT-08, EVENT-09) make no cost statement. The matrix governs desktop consequences, not cost accounting, and makes no cost statement for any event.
CA-2: FAIL — No cost model is described for any event-handling operation.
CA-3: FAIL — No reference to budget enforcement is present in the matrix. Evidence-loading and refresh events (EVENT-07, EVENT-09) do not mention whether those operations are subject to budget governance.
SR-1: PARTIAL — The matrix uses a consistent per-event table structure with named rows: Affected categories, Invalidation requirement, Refresh requirement, and per-surface rows (Topbar, Left nav, Context strip, Interaction panel, Center surfaces, Alert surface, Lower surface), plus LLM consequence, Blocking posture, Sequencing priority, and Owning doc. The nine state category labels and seven surface labels are defined in reference tables and applied consistently across all 18 event entries. However, cell contents are prose behavioral rules, not typed field values.
SR-2: PARTIAL — The table structure is consistent and stable enough that parsing code could extract per-event, per-surface consequences by row label. State category abbreviations (SCOPE, SYSTEM, STAGE, etc.) and sequencing priority values (Level 1 through Level 4) are applied consistently. However, behavioral prose in each cell is not a typed field, so parsing to structured data requires NLP rather than schema-based extraction.
AP-1: PASS — Approval requirements are explicitly documented across the matrix. EVENT-05 (Approval Written) is triggered by a persisted Class B or C approval event completing. Every event that affects approval-sensitive state specifies Class B and C gate blocking in its blocking posture row. EVENT-10 blocks all Class B and C actions during session error. The matrix treats approval gate requirements as a consistent blocking condition throughout.
AP-2: PASS — Governance gates are explicitly referenced throughout the matrix. Each event's blocking posture row specifies Class B and C gate blocking where relevant. The owning doc references per event link to desktop-state-and-context-model.md. The Cross-Event Rule "Dismiss and Acknowledge Logging" establishes governance-relevant logging for operator actions. Governance gates and action classes are the primary organizing principle of the blocking posture column.

Critical gaps (FAIL items only)
QM-2, QM-3 — No event payload or response structure defined. The matrix defines triggers and consequences in prose but specifies no input payload schema for any event and no typed response structure for any consequence. Risk: the refreshCoordinator and surface-update implementations will define independent event payload schemas without a shared contract, producing inconsistent event shapes that cannot be handled uniformly.
FR-2 — No staleness threshold defined. EVENT-08 fires when evidence has "exceeded the typical validity window" but no threshold is stated. Risk: the desktop application cannot determine when to fire EVENT-08 from this doc alone; implementations must locate and interpret the VEDA-side threshold definition independently.
FR-3 — No timestamp fields on event records or surface update payloads. The matrix requires timestamps on dismissal and acknowledgment events ("governance-relevant fact with timestamp and operator identity") but specifies no structured timestamp field on any event record or surface update payload. Risk: event timestamp formats will vary across implementation, making governance replay and session audit records inconsistent.
AL-1, AL-2, AL-3 — Activity trail logging not addressed. Several matrix events map directly to activity-trail-model.md action types: EVENT-05 (Approval Written) matches approval.decide; return trigger state transitions in EVENT-16 have governance-relevant logging consequences. The Cross-Event Rule "Dismiss and Acknowledge Logging" requires governance-relevant logging but targets a "session audit record" rather than the ecosystem activity trail, with no connection defined between the two. Risk: approval completions and governance-relevant dismissals produced through matrix events will not appear in the ecosystem activity trail despite being the most governance-critical desktop operations, breaking the cross-system audit capability.
CA-1, CA-2, CA-3 — No cost accounting. Events involving evidence API calls (EVENT-07, EVENT-08, EVENT-09) make no cost statement. Risk: evidence-loading and refresh operations triggered by matrix events have no defined budget posture; cost attribution for desktop-initiated API calls during event handling cannot be assigned from this doc.

Summary
desktop-invalidation-and-refresh-matrix.md is the most operationally detailed doc in this pass after v-forge-evidence-access-contract.md, earning 2 PASSes for explicit approval class gate blocking across every affected event and comprehensive governance gate references. The 7 PARTIALs reflect genuine partial coverage: the event table structure is consistent and navigable (SR-1, SR-2), EVENT-10 provides a partial degraded mode with recovery path (DG-1 through DG-3), and a freshness model exists for evidence events though without concrete thresholds (FR-1). The 12 FAILs concentrate in three areas: no typed event payload or response schemas (QM-2, QM-3); no staleness thresholds or timestamp fields (FR-2, FR-3); and — the most significant gap — no connection between the matrix's internal "session audit record" logging requirement and the ecosystem activity trail, despite EVENT-05 and EVENT-16 corresponding directly to action types the activity-trail-model.md requires to produce records (AL-1 through AL-3).
