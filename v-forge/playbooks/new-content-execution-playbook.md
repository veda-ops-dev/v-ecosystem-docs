# V Forge New Content Execution Playbook

## Purpose

This playbook gives operators and synced LLM sessions a practical operating sequence for executing new content work that has already been admitted into V Forge scope.

It exists to answer:

```text
How should an operator or LLM execute a new content work unit that was already approved or admitted into V Forge, create the right asset without drifting into planning, register the new structure honestly, handle publication discipline where relevant, and know when the work must stop, clarify, or return to planning?
```

This is a Tier 2 V Forge operator support document.

---

## Scope

This playbook governs:

- execution of already-admitted new content work units
- creation of new owned content assets inside bounded execution scope
- structural registration of newly created assets in the content graph
- publication-sensitive posture for new content where applicable
- stop / clarify / return decisions during new content execution

---

## Out of Scope

This playbook does not govern:

- deciding whether a new content opportunity should exist
- open-ended topic discovery or market exploration
- execution-local updates to already existing assets when no new asset is being created
- release-package work that is better modeled through the release playbook
- future experimentation or optimization programs

Those belong in Project V, VEDA, the content-update playbook, the release playbook, or later V Forge playbooks.

---

## System

- v-forge

## Core Rule

A new content execution playbook starts only after the content work unit is already inside legitimate execution scope.

The goal is not to improvise what to build.
The goal is to:
- execute the admitted asset correctly
- preserve execution truth honestly
- register the resulting structure in the content graph
- preserve publication continuity where relevant
- stop when the work reveals a planning problem instead of an execution problem

V Forge executes the new content work unit.
It does not decide whether the content should exist in the first place.

---

## When To Use This Playbook

Use this playbook when all of the following are true:
- the work concerns a new content asset that does not yet exist as execution truth
- the work unit is already admitted into V Forge through handoff or equivalent governed execution authorization
- the asset target is bounded enough to describe clearly
- the work is genuinely creation of a new asset, not merely an update to an existing one

Examples:
- create a new page that is already in approved execution scope
- execute a new content asset already named or bounded in current handoff scope
- create a new asset inside a clearly admitted content cluster or work unit
- create a new content asset whose publication posture is already understood or can be handled inside current execution discipline

---

## Do Not Use This Playbook When

Do not use this playbook when:
- the real question is whether a topic should be pursued at all
- the content target is still exploratory or undefined
- the work depends on open-ended signal gathering before asset scope can be known
- the work reveals a larger content strategy decision that has not yet been made
- the real task is primarily a release-package or surface rollout problem rather than asset creation

In those cases:
- Project V may need to decide scope
- VEDA may need to provide bounded signal first
- the release playbook may be more appropriate
- return-to-planning may be required

---

## Inputs You Should Have Before Starting

Before beginning, you should have:
- project scope clarity
- the admitted execution basis for the new content work unit
- enough specificity to know what new asset is being created
- enough current-state visibility to know whether the asset truly does not yet exist as execution truth
- any bounded constraints relevant to structure, surface, or publication

Helpful inputs may include:
- handoff package or equivalent execution authorization
- target asset description or bounded work-item statement
- surrounding graph context where the new asset will attach
- bounded execution intelligence or findings already admitted and relevant
- publication expectations if the asset is intended to go live now rather than later

If those inputs do not exist in usable form, do not pretend they do.
Clarify or stop.

## Default Operating Sequence

## Step 1 — Confirm the work is truly new-content execution

Ask:
- does the target asset not yet exist as current execution truth?
- is the work already admitted into execution?
- is the content target bounded enough to build without inventing planning on the fly?

If any of those are unclear, stop and classify correctly before acting.

### Good signs
- admitted work unit already exists
- target asset is identifiable
- the work sounds like “build this admitted asset” rather than “figure out what should exist”

### Bad signs
- the task sounds like topic exploration
- the LLM is drifting into “we should probably create a whole cluster while we’re here”
- the scope basis is vague or absent

---

## Step 2 — Inspect current execution truth first

Before creating anything, inspect what V Forge already says is true.

Check:
- whether a matching or overlapping asset already exists
- what graph context already exists around the target area
- whether adjacent assets create duplication risk
- whether existing blockers, maintenance notes, or findings already matter

Do not create new assets from assumption alone.
A surprising amount of content sprawl begins as duplicate execution caused by poor inspection.

---

## Step 3 — Restate the bounded creation scope

Restate the exact new asset in bounded terms.

Examples:
- create page X for admitted topic Y
- create new asset Z within already-approved cluster scope
- create one new content asset with specified structural role

The restated scope should make clear:
- what is being created
- what surface or asset family it belongs to
- what is explicitly not included
- whether publication is part of the current work unit or a later step

If you cannot restate the creation scope clearly, you are not ready to proceed.

---

## Step 4 — Confirm this is execution-local, not stealth planning

Ask:
- is this asset already justified by admitted scope?
- or am I now deciding that additional new assets should exist because execution discovered opportunity?

### Execution-local examples
- create the already-admitted page
- create the explicitly handed-off asset
- create the single bounded asset already named in execution scope

### Planning-relevant examples
- while creating one asset, you decide three more should exist
- the target asset only makes sense if the surrounding strategy changes first
- the asset scope now depends on unresolved topic or direction decisions

If planning relevance appears, do not keep creating under the banner of execution.
That is shadow planning.

---

## Step 5 — Create the bounded new content asset

Once scope is clear, create only the admitted asset.

During creation:
- stay inside the stated asset boundary
- preserve partial progress honestly if interrupted
- avoid absorbing adjacent ideas into the current work unit
- treat new opportunities as findings, not permissions

The new content asset should reflect the admitted execution target, not a broader speculative content program.

---

## Step 6 — Register the new structure in the content graph

A newly created asset must be registered in graph truth once the asset is real enough to be durable execution truth.

Examples:
- create the page record
- attach topic/entity/archetype relationships that now genuinely exist
- record schema usage actually applied
- record internal links actually created

Graph registration must follow what was actually built.
It must not pre-model additional hypothetical assets or future cluster structure.

If the asset was created and the graph was not updated, the work is not operationally whole.

---

## Step 7 — Decide whether publication is part of this work unit

Ask:
- is this new asset meant to remain created-but-not-published for now?
- or is publication part of the admitted execution unit?
- if publication is in scope, is authorization posture actually clear enough?

If publication is in scope:
- follow the publication-sensitive discipline from the content lifecycle and release posture
- update publication continuity honestly once the action occurs

If publication is not in scope:
- do not quietly publish because the asset exists

Asset creation is not the same as publication permission.

---

## Step 8 — Classify the outcome honestly

At the end of the work, classify the outcome truthfully.

Possible outcomes:
- completed
- partial
- blocked
- failed
- return_required

### Use completed when
- the admitted asset was created
- graph truth is aligned
- publication continuity is aligned if publication was part of the work unit

### Use partial when
- real progress occurred
- but the whole work unit did not complete

### Use blocked when
- the new asset cannot continue inside current execution bounds because of real blocker or missing condition

### Use failed when
- the attempt did not succeed

### Use return_required when
- the work revealed a planning-level contradiction, scope problem, or authorization issue that V Forge should not resolve alone

Do not classify based on optimism.
Classify based on truth.

---

## Step 9 — Record findings if the work surfaced them

If the work surfaced meaningful bounded findings, record them.

Examples:
- admitted scope was thinner than the real execution target needed
- surrounding graph context created duplication or conflict risk
- publication constraints were stronger than expected
- the new asset exposed a planning-level contradiction

A finding is not a stealth roadmap decision.
It is bounded execution truth about what was learned.

---

## Step 10 — Return to planning if needed

Return-to-planning is required when new content creation crosses execution boundaries.

Common triggers:
- the admitted asset no longer makes sense under current conditions
- the real content need is materially broader than the admitted unit
- the new asset reveals a planning contradiction or missing surrounding decision
- publication cannot proceed legitimately inside current execution bounds
- the work reveals future content needs that V Forge should not self-authorize

When returning:
- preserve what was actually created or attempted
- preserve what remains blocked or uncertain
- package bounded findings
- do not dump all execution ownership upstream

Return is correct governance, not failure theater.

---

## Fast Checks During the Work

Use these quick questions during the execution.

### Check 1
**Am I still creating the admitted asset, or am I quietly expanding the content plan?**

### Check 2
**Did the new asset become real enough that graph registration is now required?**

### Check 3
**Is publication actually in scope, or am I treating capability as permission?**

### Check 4
**If audited later, would this look like bounded execution or stealth strategy creation?**

If the answers start drifting, stop.

---

## Stop Immediately Conditions

Stop and do not continue casually when:
- the target asset is no longer clearly bounded
- the admitted scope is unclear or missing
- the new asset creation now depends on a broader strategy decision
- graph implications are no longer execution-local
- publication-sensitive action now requires stronger authorization and that posture is unclear
- the work is turning into “while we’re here, let’s create the surrounding cluster too”
- execution intelligence or signal is being treated as permission instead of input

That is the point for:
- clarification
- halt
- bounded finding
- return-to-planning

Not improvisation.

---

## Good Use of This Playbook Looks Like

Good use looks like:
- clear admitted asset scope
- inspection before creation
- no duplicate creation from assumption
- bounded creation of the intended asset only
- graph registration once the asset is real
- publication handled only if truly in scope and authorized
- honest final status
- clean return-to-planning when creation reveals a real boundary

Bad use looks like:
- inventing asset scope while building
- creating multiple adjacent assets under one admitted work unit without authority
- graph speculation ahead of real execution
- publication by convenience
- using creation work to smuggle in planning

---

## LLM Sync Reminder

A synced LLM using this playbook should remember:
- this is for new assets already admitted into execution scope
- V Forge does not decide whether the asset should exist in the first place
- graph truth follows the real built asset
- publication readiness is not publication authorization
- findings may inform planning but do not authorize new scope
- return-to-planning is correct when creation becomes a planning problem

If the LLM starts sounding like a strategist or autonomous content planner, it is out of bounds.

---

## Usage

This playbook should be used:
- by operators creating new content assets already admitted into V Forge scope
- by synced LLM sessions handling bounded new content execution
- when deciding whether new content creation stays local or must return to planning
- as the practical operator layer for already-approved new content work in V Forge
