# Project V API — Status History

## Purpose

This document defines the API contract family for Project V status-history records.

It exists to answer:

```text
How should Project V expose the historical record of status changes across governed planning records so operators and LLMs can understand what changed, when it changed, why it changed, and whether the change was legitimate without treating current status as if it appeared from nowhere?
```

This is a Tier 2 Project V API authority document.

---

## Scope

This document governs:

- the `status-history` route family in Project V
- the role of status-history records in continuity, traceability, and review
- allowed operations for listing and reading status-history records
- how status-history relates to mutable current-state records
- filtering, linkage, and visibility posture for historical status records
- validation and non-mutation posture for status-history entries

---

## Out of Scope

This document does not define:

- the full schema implementation detail for history storage
- every status vocabulary for every Project V record family
- how state transitions are approved internally by each route family
- audit record storage beyond historical status continuity
- V Forge execution-state history

Those belong in schema, status-transition, audit, or companion docs.

## Core Rule

Status-history records are continuity artifacts, not editable planning notes.

They exist so Project V can preserve a truthful record of how governed records changed over time.

A status-history record should help answer:
- what was the prior status?
- what is the new status?
- when did the change happen?
- what record changed?
- why did it change, if rationale was captured?

Status-history must not become:
- a rewriteable convenience log
- a substitute for the current canonical record
- a place to smuggle broader change narratives that belong elsewhere
- an optional afterthought that only exists when someone remembered to keep notes

---

## Record Role

A status-history record represents one status transition event for a governed Project V record.

Typical examples include:
- objective moving from draft to active
- initiative moving from proposed to approved
- work item moving from planned to in_progress or blocked
- readiness evaluation moving into a new posture
- dependency changing status after review

The historical entry exists because current status alone is not enough for continuity, audits, and LLM comprehension.

---

## Route Family

Base route family:

```text
/api/status-history
```

Canonical item route:

```text
/api/status-history/{statusHistoryId}
```

This family is project-scoped.
All operations must run inside active project scope.

---

## Supported Operations

### List
```text
GET /api/status-history
```

### Read one
```text
GET /api/status-history/{statusHistoryId}
```

No create route is exposed in first pass.
No patch route is exposed in first pass.
No delete route is exposed in first pass.

Status-history entries should normally be generated by governed state-transition logic in other route families, not authored freehand through a standalone public mutation surface.

---

## Read-Only Generation Rule

Status-history is generated as a byproduct of valid status transitions on governed records.

That means:
- route families that own status transitions create history entries when transitions succeed
- callers do not directly invent status-history records through a standalone write API
- callers do not patch old history records to improve the story later

This preserves trust.
If historical entries are directly editable, they stop being history.

---

## Record Shape

At minimum, a status-history record should expose fields shaped like:

- `id`
- `projectId`
- `recordType`
- `recordId`
- `fromStatus`
- `toStatus`
- `changedAt`
- `changeReason`
- `changedBy`
- `transitionSource`
- `createdAt`

Additional normalized or derived fields may exist in implementation, but they must remain subordinate to the governed meaning here.

---

## Field Meaning

### `recordType`
The Project V record family whose status changed.

Examples may include:
- `objective`
- `initiative`
- `work_item`
- `dependency`
- `handoff`
- `readiness_evaluation`
- `audit_record`

### `recordId`
The ID of the specific record whose status changed.

### `fromStatus`
The prior status.

### `toStatus`
The resulting status.

### `changedAt`
The timestamp of the transition event.

### `changeReason`
Optional bounded reason or transition note if that route family captures one.

### `changedBy`
Bounded actor identity or actor reference for who or what triggered the change.

### `transitionSource`
Optional classification for how the transition was triggered.

Examples may include:
- `api`
- `system_rule`
- `operator_action`
- `review_outcome`

## List Operation

### Route
```text
GET /api/status-history
```

### Purpose
List status-history entries in active project scope.

### Supported filtering should include
- `recordType`
- `recordId`
- `fromStatus`
- `toStatus`
- `changedBy`
- `transitionSource`
- date range on `changedAt`

### Sorting should support
- `changedAt`
- `createdAt`

### Includes
The route may support lightweight current-record summary includes where useful.

### Rule
List responses should make it easy to answer:
- what is the history for this work item or initiative?
- what status transitions happened recently in this project?
- who or what changed a record into its current posture?

---

## Read-One Operation

### Route
```text
GET /api/status-history/{statusHistoryId}
```

### Purpose
Return one status-history event.

### Should include
- the full history entry
- minimal linked record summary where useful

### Must not do
- mutate the meaning of the event during read
- hide current and prior status values in favor of softer prose

A history entry is a fact record about a transition.
Its job is clarity, not comfort.

---

## Relationship to Current Record State

Current record state is canonical for “what is true now.”
Status-history is canonical for “how it got here.”

Both matter.

The API family must preserve the difference:
- current record routes answer current state questions
- status-history answers transition continuity questions

Status-history must not be treated as the only source of current state.
Current-state routes must not erase the need for history.

---

## Relationship to Status Transition Rules

The allowed status transitions for each record family are governed elsewhere, especially by:
- `project-v/status-transitions.md`
- route-family-specific API docs

This route family does not define what transitions are allowed.
It defines how valid transitions are preserved historically once they occur.

---

## Non-Mutation Rule

History entries should not be patchable in first pass.

If implementation needs bounded repair posture for rare operational errors, that must be handled through governed administrative repair logic, not a casual general PATCH route.

Reason:
status history is valuable only if later operators and LLMs can trust it as preserved transition continuity.

---

## Change Reason Rule

`changeReason` is optional bounded context, not a place for giant narratives.

Good uses:
- “Approved in readiness review”
- “Blocked after dependency failed evaluation”
- “Superseded by revised initiative plan”

Bad uses:
- full audit report pasted inline
- broad planning essay
- chat transcript excerpt dump

Where broader explanation is required, it should live in the right companion artifact.

---

## Validation Rules

### Structural validation → 400
Examples:
- malformed UUID on read/list filter inputs
- malformed date filter
- unknown query parameter

### Semantic validation → 422
Examples:
- invalid `recordType` filter
- invalid status filter values
- invalid transition-source filter value
- invalid date-range semantics

### Scope failure → 404
Examples:
- history entry absent in project scope
- linked record absent in project scope where item read includes context
- history entry exists elsewhere but is out of scope

Conflict posture is usually not central to this route family because it is read-only in first pass.

---

## Generation Expectations

The Project V implementation should strongly prefer automatic generation of status-history entries whenever a governed status transition succeeds.

That means:
- no legitimate transition should happen silently
- no transition should update current state while skipping continuity recording
- history records should be timestamped and project-scoped consistently

A system that tracks status changes only sometimes is not preserving continuity honestly.

---

## LLM and Operator Use Rule

A capable LLM or operator should be able to use this route family to:
- reconstruct the transition path of a record
- understand why a record reached its current state
- separate stale assumptions from actual historical transitions
- review recent movement across important planning records

They should not need to infer the whole story from scattered timestamps and guesswork.

---

## Hammer Expectations

Project V hammer should verify at minimum that:
- valid status transitions produce history entries reliably
- invalid transitions do not produce fake history entries
- list filtering by record type/id and date range works
- out-of-scope history access fails non-leakily
- current-state mutation without history generation does not occur in governed flows
- history entries are not writable through this route family in first pass
